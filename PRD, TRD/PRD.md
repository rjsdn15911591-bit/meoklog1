## 1. 제품 개요

### 1.1 제품 한 줄 정의

> **음식 사진 한 장으로 칼로리·영양소를 AI가 자동 분석하고, 그룹원들과 하루 식단을 실시간 비교·공유할 수 있는 그룹형 AI 식단 관리 웹 앱.**

### 1.2 배경

기존 식단 기록 앱의 사용 장벽은 크게 두 가지다. 첫째, 매 식사마다 음식명을 검색하고 칼로리를 수동 입력하는 번거로움. 둘째, 혼자 기록하는 구조로 인한 지속 동기 부족. 먹로그는 AI 사진 분석으로 입력 마찰을 제거하고, 그룹 공유·비교로 지속 동기를 부여한다.

### 1.3 참고 서비스

| 서비스 | 참고 요소 | 먹로그와의 차이 |
|--------|----------|----------------|
| 셋로그(Setlog) | 그룹 방 시스템, 실시간 피드 구조, 초대 코드 방식 | 영상 대신 음식 사진 + AI 칼로리 분석 |
| 다이어트 앱 (MyFitnessPal 등) | 칼로리 DB, 목표 설정 | 사진 기반 자동 분석, 그룹 비교 없음 |
| 인스타그램 | 피드·댓글·이모티콘 반응 | 프라이빗 그룹 구조, 데이터 중심 |

## 2. 문제 정의

### 2.1 핵심 문제

```
문제 1. 식단 기록의 입력 마찰
  현상: 기존 앱은 식사마다 음식명 검색 → 양 입력 → 칼로리 확인 과정이 필요.
  결과: 사용자의 72%가 2주 내 이탈 (업계 평균).
  해결: 사진 한 장 업로드 → AI 자동 분석으로 전체 흐름을 1단계로 압축.

문제 2. 지속 동기 부족
  현상: 혼자 하는 식단 기록은 목표 이탈 시 재개 계기가 없음.
  결과: 장기 사용률 낮음.
  해결: 그룹 실시간 비교로 사회적 동기(경쟁·공유·반응) 부여.

문제 3. AI 인식 오류 수용성
  현상: 100% 정확한 AI 식품 인식은 불가능.
  결과: 오인식 시 사용자 신뢰 하락.
  해결: AI 결과를 사용자가 항상 수정 가능하게 하고, "예상 칼로리"로 명확히 표현.
```

### 2.2 해결 방향

```
사진 촬영/업로드
    ↓ (1단계)
AI 자동 분석 (음식 인식 + 칼로리 + 탄단지)
    ↓ (2단계, 선택)
사용자 결과 확인 및 수정
    ↓ (3단계)
개인 로그 저장 + 그룹 피드 공유
    ↓ (자동)
그룹원 실시간 비교 반영
```

## 3. 목표 사용자

### 3.1 1차 타겟 (Primary User)

**페르소나 A — 다이어트 도전 대학생**

```
나이:       20~25세
상황:       학기 중 식단 관리를 시도하지만 지속이 어려움
동기:       친구들과 함께 하면 더 잘 될 것 같다는 기대
행동 패턴:  스마트폰으로 하루 10회 이상 SNS 확인, 사진 공유에 익숙
기대:       복잡한 입력 없이 사진만 찍으면 되는 편리함
            친구가 뭘 먹는지 보면서 자극받는 재미
불안:       AI가 엉뚱한 음식으로 인식할까봐 걱정
```

**페르소나 B — 헬스 목표 직장인**

```
나이:       25~32세
상황:       증량 또는 감량 목표로 탄단지 비율을 관리하고 싶음
동기:       정확한 영양소 데이터 트래킹
행동 패턴:  헬스 커뮤니티 활동, 인바디 주기적 측정
기대:       탄수화물/단백질/지방 비율 자동 계산
            팀원/친구와 식단 목표 공유
불안:       칼로리 수치 정확도에 민감
```

### 3.2 2차 타겟 (Secondary User)

- 가족 단위 건강 관리: 부모-자녀 그룹으로 서로 식단 확인
- 다이어트 챌린지 그룹: 온라인 커뮤니티 기반 그룹 도전
- 수업 과제 시연: AI와 머신러닝 수업 프로젝트 데모 대상

### 3.3 타겟 외 사용자 (Out of Scope)

- 의료 목적 임상 식단 관리가 필요한 환자
- 전문 영양사·트레이너의 클라이언트 관리 도구
- 어린이 사용자

## 4. 제품 목표 & 성공 지표

### 4.1 Phase 1 (MVP) 목표

| 목표 | 측정 지표 | 달성 기준 | 상태 |
|------|----------|----------|------|
| 핵심 루프 완성 | 사진 업로드 → AI 분석 → 그룹 피드 반영 전 흐름 동작 | 에러 없이 E2E 완료 | ✅ |
| AI 분석 정확도 | GPT-4o Vision 칼로리 추정 오차율 | 실제 칼로리 대비 ±20% 이내 | ✅ |
| 실시간 반영 | 피드 업로드 후 그룹원 화면 반영 | 5초 이내 | ✅ |
| 사용 편의성 | 업로드 ~ 저장 완료 | 3단계 이하 (사진 선택 → 카테고리 → 저장) | ✅ |

### 4.2 Phase 2 이후 목표 (참고)

- 7일 연속 사용률 40% 이상
- 그룹 생성 후 3일 이내 멤버 2명 이상 초대율 60%
- AI 분석 결과 수정 없이 저장하는 비율 50% 이상 (정확도 신뢰 지표)

## 6. 비기능 요구사항

### 6.1 성능

| 항목 | 기준 |
|------|------|
| AI 분석 응답 시간 | 사진 업로드 후 분석 결과 표시까지 **5초 이내** |
| 그룹 피드 실시간 반영 | 업로드 후 그룹원 피드 반영까지 **5초 이내** |
| 페이지 초기 로드 | LCP(Largest Contentful Paint) **3초 이내** |
| API 응답 시간 | 일반 CRUD 엔드포인트 **500ms 이내** |
| 이미지 업로드 | 10MB 이하 JPEG/PNG/WEBP 지원 |

### 6.2 보안

| 항목 | 기준 |
|------|------|
| 인증 | Google OAuth 2.0 + JWT (만료 7일) |
| 데이터 접근 | Supabase RLS — 본인 데이터만 읽기/쓰기, 그룹원 데이터는 읽기만 |
| HTTPS | 모든 통신 TLS 1.2 이상 강제 |
| 이미지 저장 | Cloudinary CDN — 업로더 본인만 삭제 가능 |
| 민감 정보 | 신체 정보(키·몸무게 등)는 암호화 전송, DB 저장 시 일반 컬럼 허용 (개인 앱 수준) |

### 6.3 접근성

| 항목 | 기준 |
|------|------|
| 색상 대비 | WCAG AA 기준 4.5:1 이상 (본문 텍스트) |
| 터치 타겟 | 최소 44×44px (WCAG AAA 권장) |
| 반응형 | 모바일(375px~) / 태블릿(768px~) / 데스크탑(1024px~) |
| 이미지 alt | 모든 음식 사진에 alt 텍스트 자동 생성 |

### 6.4 호환성

| 항목 | 기준 |
|------|------|
| 브라우저 | Chrome 110+, Safari 16+, Edge 110+, Firefox 110+ |
| 모바일 OS | iOS 15+, Android 11+ (모바일 브라우저) |
| 화면 해상도 | 360px ~ 1440px |

### 6.5 가용성 & 안정성

| 항목 | 기준 |
|------|------|
| 서비스 가용성 | 99% uptime 목표 (Phase 1 MVP 기준, SLA 미적용) |
| AI 모델 실패 처리 | 분석 실패 시 수동 입력 폼으로 자동 전환 |
| 이미지 업로드 실패 | 재시도 UI 제공, 에러 메시지 명확히 표시 |

## 7. 사용자 흐름

### 7.1 신규 사용자 온보딩 흐름

```
앱 최초 접속
    │
    ▼
로그인 화면 ─→ Google 로그인 버튼 탭
    │
    ▼
Google OAuth 인증 완료
    │
    ▼
신규 사용자 여부 확인
    ├─ 신규 → 신체 정보 입력 화면 (이름/나이/성별/키/몸무게/활동량/목표)
    │              ↓
    │         목표 칼로리 자동 계산 & 확인
    │
    └─ 기존 → 카메라 탭으로 바로 이동
    │
    ▼
카메라 탭 (홈)
```

### 7.2 핵심 루프 — 음식 기록 & 그룹 공유

```
카메라 탭 진입
    │
    ▼
사진 촬영 또는 갤러리에서 선택
    │
    ▼
식사 카테고리 선택 (시간 기반 자동 추천 → 사용자 확인)
    │
    ▼
서버로 업로드 + AI 분석 시작
    │
    ▼ (분석 중 로딩 표시 — lavender 카드)
    │
    ▼
AI 분석 결과 표시
    ├─ 결과 이상 없음 → 저장하기 탭
    └─ 수정 필요     → 음식명/양/칼로리 수정 → 저장하기 탭
    │
    ▼
개인 로그에 저장 + 그룹 피드에 자동 공유
    │
    ▼
그룹원들 실시간 피드 반영
그룹원들 비교 탭 데이터 실시간 갱신
```

### 7.3 그룹 참가 흐름

```
그룹 탭 진입
    │
    ▼
그룹 없음 화면
    ├─ [+ 새 그룹 만들기] 탭
    │       ↓
    │   그룹명 입력 → 생성
    │       ↓
    │   8자리 코드 발급 화면 (복사 버튼)
    │
    └─ [그룹 코드로 참가] 탭
            ↓
        코드 8자리 입력
            ↓
        코드 유효 확인 → 그룹 참가 완료
    │
    ▼
그룹 피드 화면 (오늘 식단 실시간 피드)
```

#### SCR-02 카메라 탭 — 4단계 스텝 플로우

```
[Step 1: 사진 선택]
  - 사진 드래그 앤 드롭 영역 (중앙)
  - [카메라 촬영] 버튼 (peach 배경)
  - [갤러리 선택] 버튼 (secondary)

[Step 2: 카테고리 선택]
  - 아침/점심/저녁/간식 pill 탭 (시간 기반 자동 활성)
  - 사진 미리보기 (상단 썸네일)

[Step 3: AI 분석 중]
  - lavender AccentCard + 로딩 스피너
  - "AI가 음식을 분석하고 있어요" (font-kedu)
  - 진행 퍼센트 표시

[Step 4: 결과 확인 & 저장]
  - 사진 미리보기
  - 음식 목록 (음식명 / 양 / 칼로리) — 각 행 수정 버튼
  - + 음식 추가 텍스트 버튼

  - [저장하기] 버튼 (peach, full-width)
```

#### SCR-03 로그 탭

## 9. 제약사항 & 가정

### 9.1 기술 제약사항

```
- AI 음식 인식은 Food-101 데이터셋 기반이므로 한국 음식 인식률이 낮을 수 있음.
  → 한국 음식 보완 데이터 추가 학습 필요 (Phase 2)
- 브라우저에서 카메라 직접 접근은 HTTPS 환경에서만 동작.
  → 로컬 개발 시 127.0.0.1 또는 HTTPS 터널링 필요
- Cloudinary 무료 티어: 월 25GB 저장, 25GB 대역폭 제한.
  → MVP 단계에서는 충분, 사용자 증가 시 플랜 업그레이드 필요
- Supabase 무료 티어: DB 500MB, Realtime 200 동시 접속.
  → MVP 단계에서는 충분
```

### 9.2 제품 가정

```
- 사용자는 스마트폰 또는 PC 웹 브라우저로 접근한다.
- 사용자는 Google 계정을 보유하고 있다.
- 그룹 기능은 지인 기반(친구/가족/팀)으로 사용된다. (공개 커뮤니티 아님)
- AI 분석 결과는 "예상값"으로 제공되며, 사용자가 수정 가능한 구조를 유지한다.
- 하루 기준은 04:00으로 고정하며, 사용자가 변경할 수 없다. (Phase 3 고려)
```
